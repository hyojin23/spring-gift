# ADR-0003: 서비스 계층 트랜잭션 경계 명시

## 상태

Accepted

## 맥락

서비스 계층에는 조회, 생성, 수정, 삭제 메서드가 섞여 있다. 일부 서비스는 데이터베이스에 접근하고, 일부 서비스는 카카오 API 같은 외부 시스템 호출을 담당한다.

트랜잭션 경계가 명확하지 않으면 다음 문제가 생길 수 있다.

- 조회 메서드와 쓰기 메서드의 의도가 코드에서 잘 드러나지 않는다.
- 외부 API 호출이 불필요하게 DB 트랜잭션 안에 포함될 수 있다.
- 새로운 서비스나 메서드를 추가할 때 트랜잭션 적용 기준이 달라질 수 있다.
- 리팩토링 과정에서 기능 변경 없이 경계만 정리했는지 검증하기 어렵다.

고려한 선택지는 다음과 같다.

1. 모든 서비스 메서드에 개별적으로 `@Transactional` 또는 `@Transactional(readOnly = true)`를 붙인다.
2. DB 접근 서비스에는 클래스 레벨로 `@Transactional(readOnly = true)`를 두고, 쓰기 메서드에만 `@Transactional`을 명시한다.
3. 별도 트랜잭션 선언 없이 Spring Data JPA repository 동작에 의존한다.

## 결정

DB 접근 서비스는 클래스 레벨에 `@Transactional(readOnly = true)`를 기본으로 둔다. 생성, 수정, 삭제처럼 데이터를 변경하는 메서드는 메서드 레벨에 `@Transactional`을 명시한다.

외부 API 호출 중심 서비스에는 DB 트랜잭션을 추가하지 않는다.

적용 규칙:

1. 조회 중심 메서드는 클래스 레벨 `readOnly = true` 기본값을 사용한다.
2. 생성, 수정, 삭제 메서드는 메서드 레벨 `@Transactional`로 쓰기 트랜잭션을 명시한다.
3. 카카오 인증, 카카오 메시지 발송처럼 외부 API 호출 중심 서비스에는 트랜잭션을 적용하지 않는다.
4. 트랜잭션 경계 정리는 동작 변경을 목적으로 하지 않으며, 기존 API 응답과 관리자 화면 흐름을 유지한다.

## 이유

서비스 계층의 트랜잭션은 반복적으로 따라야 하는 경계다. 한 번 기준이 흐려지면 이후 서비스마다 다른 방식으로 트랜잭션이 선언될 수 있다.

클래스 레벨 `readOnly = true`를 기본으로 두면 조회 메서드가 많은 서비스에서 의도를 간결하게 표현할 수 있다. 반대로 쓰기 메서드는 메서드 레벨 `@Transactional`을 통해 데이터 변경이 발생한다는 점을 명확히 드러낼 수 있다.

외부 API 호출 서비스는 DB 트랜잭션과 묶지 않는다. 네트워크 호출을 트랜잭션 안에 넣으면 트랜잭션 시간이 길어지고, DB 작업과 외부 시스템 호출의 책임이 섞인다.

## 결과

장점:

- 조회와 쓰기 메서드의 트랜잭션 의도가 코드에서 드러난다.
- 서비스 계층의 트랜잭션 적용 기준이 일관된다.
- 외부 API 호출이 불필요하게 DB 트랜잭션에 포함되는 것을 막을 수 있다.
- 기능 변경 없이 경계만 정리했는지 테스트로 확인하기 쉽다.

단점:

- 쓰기 메서드에 `@Transactional`을 누락하면 클래스 레벨 `readOnly = true`가 적용될 수 있다.
- 모든 메서드에 개별 annotation을 붙이는 방식보다 메서드 단위 의도는 덜 직접적으로 보일 수 있다.
- 서비스가 DB 접근과 외부 API 호출을 함께 수행하면 추가 분리가 필요할 수 있다.

## 대안

### 모든 메서드에 개별 annotation 부여

각 메서드에 `@Transactional` 또는 `@Transactional(readOnly = true)`를 직접 붙이는 방식도 고려했다.

이 방식은 메서드 하나만 보아도 트랜잭션 의도가 명확하다. 다만 조회 메서드가 많은 서비스에서는 반복이 늘어나고, 같은 기준을 여러 곳에 계속 작성해야 한다.

### Repository 트랜잭션에 의존

서비스 계층에 별도 트랜잭션을 선언하지 않고 repository 동작에 의존하는 방식도 고려했다.

이 방식은 코드가 가장 적다. 다만 유스케이스 단위의 트랜잭션 경계가 드러나지 않고, 여러 repository 호출을 하나의 작업으로 묶어야 하는 경우 서비스 계층의 책임을 표현하기 어렵다.

## 적용 범위

- DB 접근을 수행하는 서비스 계층
- 상품, 옵션, 회원, 주문, 위시, 카테고리 관련 서비스
- 조회 메서드와 생성, 수정, 삭제 메서드의 트랜잭션 선언

카카오 로그인 클라이언트, 카카오 메시지 발송처럼 외부 API 호출이 중심인 컴포넌트에는 이 결정을 적용하지 않는다.
