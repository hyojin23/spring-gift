# ADR-0005: 주문 생성 시 옵션 재고 차감에 비관적 락 사용

## 상태

Accepted

## 맥락

주문 생성 과정에서는 옵션 재고 차감, 회원 포인트 차감, 주문 저장, 위시 정리가 하나의 트랜잭션 안에서 수행된다.

현재 옵션 재고 차감 흐름은 다음과 같다.

1. 주문할 옵션을 조회한다.
2. 옵션 엔티티의 수량을 메모리에서 차감한다.
3. 변경된 옵션을 저장한다.
4. 주문을 저장한다.

이 흐름에서 같은 옵션에 대한 주문이 동시에 들어오면 여러 트랜잭션이 같은 재고 값을 읽고 각각 차감한 뒤 저장할 수 있다. 이 경우 실제 주문 수량보다 재고가 적게 차감되는 lost update 문제가 발생할 수 있다.

고려한 선택지는 다음과 같다.

1. 옵션 조회 시 비관적 쓰기 락을 사용해 같은 옵션 주문을 순차 처리한다.
2. `@Version` 기반 낙관적 락을 사용하고 충돌 시 주문을 실패 처리한다.
3. `quantity >= 요청 수량` 조건을 포함한 update 쿼리로 재고를 원자적으로 차감한다.
4. 주문 요청 앞단에 메시지 큐를 두고 같은 옵션 주문을 순차 처리한다.

## 결정

주문 생성 시 옵션을 조회할 때 `PESSIMISTIC_WRITE` 락을 사용한다.

예상 흐름:

1. `OrderService#createOrder`가 주문 트랜잭션을 시작한다.
2. `OptionRepository`에서 주문할 옵션을 비관적 쓰기 락으로 조회한다.
3. 같은 옵션에 대한 다른 주문 트랜잭션은 락이 해제될 때까지 대기한다.
4. 락을 획득한 트랜잭션은 최신 재고 기준으로 수량 차감 가능 여부를 판단한다.
5. 주문 트랜잭션이 커밋되면 다음 대기 트랜잭션이 최신 재고를 읽고 처리한다.

## 이유

주문은 인기 상품이나 특정 옵션에 요청이 몰릴 수 있는 흐름이다. 충돌이 자주 발생할 수 있는 상황에서 낙관적 락을 사용하면 여러 주문이 동시에 진행된 뒤 커밋 시점에 반복적으로 실패할 수 있다.

비관적 락은 같은 옵션 row에 대한 주문을 DB 수준에서 순차 처리한다. 따라서 각 주문은 최신 재고를 기준으로 차감 여부를 판단할 수 있고, 재고가 부족한 경우 기존 재고 부족 예외로 실패시킬 수 있다.

메시지 큐를 사용하면 순차 처리와 확장성을 더 명확하게 설계할 수 있지만, 현재 프로젝트 범위에서는 별도 인프라와 비동기 주문 처리 정책이 필요해 변경 범위가 크다. 이번 결정은 현재 동기 주문 생성 흐름을 유지하면서 재고 차감의 정합성을 보장하는 데 집중한다.

또한 주문 생성 후 카카오 알림은 `AFTER_COMMIT` 이벤트로 분리되어 있으므로, 옵션 row lock을 잡은 트랜잭션 안에서 외부 API를 직접 호출하지 않는다.

## 결과

장점:

- 같은 옵션에 대한 주문을 DB row 단위로 순차 처리할 수 있다.
- 주문마다 최신 재고를 기준으로 차감 가능 여부를 판단할 수 있다.
- 재고 차감, 포인트 차감, 주문 저장을 하나의 트랜잭션 안에서 유지할 수 있다.
- 메시지 큐를 도입하지 않고 현재 동기 주문 흐름 안에서 정합성을 개선할 수 있다.

단점:

- 같은 옵션에 주문이 몰리면 락 대기 시간이 늘어날 수 있다.
- 트랜잭션이 길어지면 락 보유 시간이 길어진다.
- 데드락이나 락 타임아웃 가능성을 고려해야 한다.
- DB 락에 의존하므로 테스트는 실제 트랜잭션 기반 통합 테스트로 검증해야 한다.

## 대안

### 낙관적 락

`Option`에 `@Version` 필드를 추가하고, 커밋 시점에 version 충돌을 감지하는 방식도 고려했다.

이 방식은 충돌이 드문 상황에서는 락 대기 비용이 적다. 다만 주문은 같은 옵션에 요청이 몰릴 수 있고, 충돌이 잦으면 주문 실패와 재시도가 반복될 수 있다. 현재 주문 흐름에서는 사용자 경험과 재시도 정책을 별도로 설계해야 하므로 이번 결정에서는 제외했다.

### 조건부 update 쿼리

`update options set quantity = quantity - :amount where id = :id and quantity >= :amount` 형태의 쿼리로 재고를 원자적으로 차감하는 방식도 고려했다.

이 방식은 재고 차감만 보면 효율적이다. 다만 현재는 `Option` 도메인 객체의 `subtractQuantity` 메서드가 차감 수량과 재고 부족 검증을 담당하고 있으므로, update 쿼리 방식은 도메인 메서드를 우회한다. 이번 변경에서는 도메인 검증 흐름을 유지하는 비관적 락 방식을 선택했다.

### 메시지 큐

주문 요청을 메시지 큐에 넣고 같은 옵션 주문을 순차 소비하는 방식도 고려했다.

이 방식은 대규모 트래픽과 비동기 처리에 적합하다. 다만 현재 API는 주문 요청에 대해 즉시 주문 결과를 반환하는 동기 흐름이며, 메시지 큐를 도입하면 주문 상태, 실패 처리, 재시도, 사용자 응답 정책까지 함께 설계해야 한다. 현재 과제 범위에서는 변경 비용이 크다.

## 적용 범위

- `OrderService#createOrder`
- `OptionRepository`의 주문용 옵션 조회
- 옵션 재고 차감 동시성 제어
- 주문 생성 동시성 통합 테스트

관리자 옵션 수정, 상품 옵션 조회, 메시지 발송 비동기화에는 이 결정을 적용하지 않는다.
