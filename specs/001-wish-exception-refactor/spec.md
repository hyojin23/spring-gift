# 기능 명세서: Wish 패키지 예외 처리 리팩토링

**Feature Branch**: `001-wish-exception-refactor`  
**작성일**: 2026-05-16  
**상태**: 초안  
**입력**: "예외 처리 기능을 리팩토링 한다. category 패키지에 되어 있는 것처럼 wish 패키지도 똑같은 구조로 리팩토링 해야 한다."

## 사용자 시나리오 및 테스트 *(필수)*

### 사용자 시나리오 1 - Wish 도메인 예외의 표준화된 응답 (우선순위: P1)

Wish 기능에서 도메인 예외가 발생할 때 시스템은 `category` 패키지와 같은 방식으로 일관된 HTTP 에러 응답을 반환해야 합니다.

**우선순위 이유**: Wish API 소비자는 예측 가능한 에러 응답을 받아야 하며, 예외 처리 방식의 일관성은 유지보수성과 디버깅 효율을 높입니다.

**독립적 테스트**: Wish 도메인 예외를 트리거하고 표준화된 HTTP 상태와 JSON 에러 형식을 검증합니다.

**승인 시나리오**:

1. **Given** 없는 위시 아이템을 조회할 때, **When** 클라이언트가 요청하면, **Then** 응답은 HTTP 404와 함께 표준 에러 JSON을 반환합니다.
2. **Given** 다른 사용자의 위시 아이템에 접근 시도할 때, **When** 권한이 없으면, **Then** 응답은 HTTP 403과 함께 표준 에러 JSON을 반환합니다.
3. **Given** 인증 정보가 없거나 유효하지 않을 때, **When** 클라이언트가 요청하면, **Then** 응답은 HTTP 401과 함께 표준 에러 JSON을 반환합니다.

---

### 사용자 시나리오 2 - Category와 동일한 패키지 구조 적용 (우선순위: P2)

Wish 패키지의 예외 처리 구조를 `category` 패키지와 동일한 형태로 맞춰, 팀이 일관된 예외 작성 및 관리 패턴을 사용할 수 있도록 합니다.

**우선순위 이유**: 일관된 구조는 코드 이해도를 높이고, 새로운 예외를 추가할 때 발생하는 중복과 혼선을 줄입니다.

**독립적 테스트**: 코드 리뷰를 통해 Wish 예외 클래스 위치, 명명 규칙, 핸들러 통합 방식이 category 패키지와 일치하는지 확인합니다.

**승인 시나리오**:

1. Wish 도메인 예외 클래스가 `gift.wish.exception` 패키지 아래에 위치합니다.
2. Wish 예외는 중앙 집중식 exception 핸들러에 의해 처리됩니다.

---

### 사용자 시나리오 3 - 기존 Wish 기능 동작 유지 (우선순위: P3)

리팩토링 후에도 기존 Wish 기능의 모든 동작과 API 계약이 변경되지 않아야 합니다.

**우선순위 이유**: 기존 API 사용자와의 호환성을 유지하는 것은 필수입니다.

**독립적 테스트**: 현재 Wish 관련 통합 테스트를 실행하여 회귀 여부를 확인합니다.

**승인 시나리오**:

1. 기존 Wish API의 에러 응답 HTTP 상태가 변경되지 않습니다.

---

### 엣지 케이스

- 삭제된 위시 아이템 조회 시 적절한 404 응답을 반환합니다.- 인증 정보가 없거나 유효하지 않은 경우 401 응답을 반환합니다.- 권한 없는 사용자의 위시 수정/삭제 시도는 403 응답을 반환합니다.
- 매핑되지 않은 런타임 예외는 표준 에러 형식으로 처리되어 HTTP 500을 반환합니다.

## 요구사항 *(필수)*

### 기능 요구사항

- **FR-001**: Wish 도메인 예외를 `gift.wish.exception` 패키지로 이동하거나 재구성합니다.
- **FR-002**: `WishNotFoundException`, `UnauthorizedWishAccessException` 등의 Wish 예외 클래스를 작성합니다.
- **FR-003**: Wish 예외를 중앙 집중식 exception 핸들러에서 일관된 JSON 응답으로 변환합니다.
- **FR-004**: Wish 컨트롤러에서 중복된 예외 처리 로직을 제거하고 예외 발생 흐름으로 통일합니다.
- **FR-005**: Wish 예외 처리 리팩토링을 검증하는 단위 및 통합 테스트를 추가합니다.
- **FR-006**: Wish API 문서에 새 에러 응답 형식을 명시합니다.

### 주요 엔티티

- **WishNotFoundException**: 요청한 위시 아이템을 찾을 수 없을 때 발생합니다.
- **UnauthorizedWishAccessException**: 다른 사용자의 위시에 접근하려 할 때 발생합니다.
- **WishException**: Wish 도메인 예외의 공통 기본 유형입니다.
- **ErrorResponse**: 클라이언트에 반환되는 표준화된 에러 페이로드입니다.
- **GlobalExceptionHandler**: 예외를 HTTP 상태 및 `ErrorResponse`로 변환하는 중앙 처리 컴포넌트입니다.

## 성공 기준 *(필수)*

### 측정 가능한 결과

- **SC-001**: 리팩토링 후 모든 기존 Wish 통합 테스트가 통과합니다.
- **SC-002**: Wish 도메인 예외가 모두 표준화된 JSON 에러 형식으로 응답합니다.
- **SC-003**: Wish 컨트롤러에서 개별 예외 처리 로직이 제거되어 중앙 처리로 일관화됩니다.
- **SC-004**: Category와 Wish 패키지 간 예외 처리 구조 일관성이 코드 리뷰로 확인됩니다.

## 가정사항

- Category 패키지의 예외 처리 구조가 이미 완성되어 있고, Wish 패키지는 이를 동일하게 따릅니다.
- 중앙 집중식 exception 핸들러(예: global controller advice)가 이미 존재하거나 이번 작업에서 사용 가능합니다.
- Wish 패키지의 현재 컨트롤러는 예외 처리 로직이 분산되어 있습니다.
- 기존 Wish API 사용자를 위해 HTTP 상태 코드는 변경되지 않습니다.

---

**명세서 주석**: Wish 패키지 외의 다른 도메인 예외 처리 리팩토링(예: product, member, order)은 별도 feature로 처리합니다.
