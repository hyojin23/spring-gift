# 기능 명세서: Option 삭제 검증 조회 최적화

**Feature Branch**: `003-option-deletion-validation-refactor`  
**작성일**: 2026-05-17  
**상태**: 초안  
**입력**: "삭제 가능 여부 검증을 count 쿼리로 바꾸기"

## 사용자 시나리오 및 테스트 *(필수)*

### 사용자 시나리오 1 - 마지막 옵션 삭제 검증의 조회 비용 개선 (우선순위: P1)

Option 삭제 시 시스템은 상품의 마지막 옵션 여부를 판단해야 합니다. 현재 구현은 상품의 옵션 목록 전체를 조회한 뒤 개수를 확인할 수 있으므로, 삭제 가능 여부 판단에 필요한 개수만 조회하도록 개선합니다.

**우선순위 이유**: 삭제 가능 여부 검증은 옵션 목록의 상세 데이터가 아니라 옵션 개수만 필요합니다. count query를 사용하면 의도가 명확해지고 불필요한 엔티티 조회를 줄일 수 있습니다.

**독립적 테스트**: `OptionService.deleteOption`에서 상품의 옵션 개수가 1개 이하이면 기존처럼 `OptionDeletionNotAllowedException`이 발생하고, 2개 이상이면 삭제 대상 옵션 조회 및 삭제 흐름으로 진행되는지 검증합니다.

**승인 시나리오**:

1. **Given** 상품에 옵션이 1개만 있을 때, **When** 해당 옵션 삭제를 요청하면, **Then** 기존과 동일하게 마지막 옵션 삭제 제한 예외가 발생합니다.
2. **Given** 상품에 옵션이 2개 이상 있을 때, **When** 존재하지 않는 옵션 삭제를 요청하면, **Then** 마지막 옵션 제한이 아니라 옵션 미존재 예외가 발생합니다.
3. **Given** 상품에 옵션이 2개 이상 있고 삭제 대상 옵션이 존재할 때, **When** 삭제를 요청하면, **Then** 기존과 동일하게 옵션이 삭제됩니다.

---

### 사용자 시나리오 2 - 기존 API 계약 유지 (우선순위: P2)

삭제 검증 조회 방식은 내부 구현만 변경하며, 외부 API의 HTTP status, error code, message, 성공 응답은 변경하지 않습니다.

**우선순위 이유**: 이 작업은 성능과 코드 의도 개선을 위한 리팩토링이며, 클라이언트가 체감하는 동작을 바꾸면 안 됩니다.

**독립적 테스트**: 기존 Option controller/service 테스트가 동일하게 통과하는지 확인합니다.

**승인 시나리오**:

1. 마지막 옵션 삭제 요청은 기존처럼 HTTP 400과 `OPTION.DELETE_NOT_ALLOWED` 응답으로 변환됩니다.
2. 정상 삭제 요청은 기존처럼 HTTP 204를 반환합니다.
3. 존재하지 않는 상품 또는 옵션에 대한 응답 의미는 변경되지 않습니다.

---

### 엣지 케이스

- 상품이 존재하지 않으면 count query를 실행하기 전에 기존처럼 상품 미존재 예외가 발생합니다.
- 상품에 옵션이 0개로 잘못된 데이터 상태가 있어도 삭제 제한 예외가 발생해야 합니다.
- 상품에 옵션이 2개 이상이면 마지막 옵션 삭제 제한 검증을 통과해야 합니다.
- 삭제 대상 옵션이 요청한 상품에 속하지 않으면 기존처럼 옵션 미존재 예외가 발생해야 합니다.

## 요구사항 *(필수)*

### 기능 요구사항

- **FR-001**: `OptionRepository`는 상품별 옵션 개수를 조회하는 repository method를 제공해야 합니다.
- **FR-002**: `OptionService`의 마지막 옵션 삭제 검증은 옵션 목록 전체 조회 대신 상품별 옵션 개수 조회를 사용해야 합니다.
- **FR-003**: 마지막 옵션 삭제 제한 정책은 기존과 동일하게 유지해야 합니다.
- **FR-004**: Option 삭제 API의 성공/실패 HTTP 응답 계약은 변경하지 않아야 합니다.
- **FR-005**: 삭제 검증 흐름을 검증하는 단위 테스트 또는 기존 테스트를 count query 기반으로 갱신해야 합니다.

### 주요 엔티티

- **OptionRepository**: 상품별 옵션 목록 조회와 개수 조회를 제공하는 JPA repository입니다.
- **OptionService**: 옵션 삭제 시 상품 존재 여부, 마지막 옵션 삭제 제한, 삭제 대상 옵션 존재 여부를 검증합니다.
- **OptionDeletionNotAllowedException**: 상품에 남은 옵션이 1개 이하라 삭제할 수 없을 때 발생하는 도메인 예외입니다.

## 성공 기준 *(필수)*

### 측정 가능한 결과

- **SC-001**: `OptionService.validateCanDelete`는 `findByProductId(productId).size()` 방식 대신 count query 기반으로 동작합니다.
- **SC-002**: 마지막 옵션 삭제 제한 테스트가 통과합니다.
- **SC-003**: 존재하지 않는 옵션 삭제 테스트가 count query 기반 stubbing으로 통과합니다.
- **SC-004**: `./gradlew test --tests *Option*`이 통과합니다.
- **SC-005**: 외부 API 응답 계약의 변경이 없습니다.

## 가정사항

- Option 예외 처리 구조는 `002-option-exception-refactor`에서 이미 완료되었습니다.
- 이번 작업은 삭제 가능 여부 검증의 내부 조회 방식만 변경합니다.
- 데이터베이스 schema 변경은 필요하지 않습니다.
- product, order 등 다른 도메인의 리팩토링은 이번 범위에 포함하지 않습니다.
