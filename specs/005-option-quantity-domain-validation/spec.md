# 기능 명세서: Option 수량 도메인 검증 강화

**Feature Branch**: `005-option-quantity-domain-validation`  
**작성일**: 2026-05-17  
**상태**: 초안  
**입력**: "Option 도메인 검증 강화"

## 사용자 시나리오 및 테스트 *(필수)*

### 사용자 시나리오 1 - Option 생성 시 수량 불변식 보장 (우선순위: P1)

Option은 수량을 가진 도메인 객체이며, 생성된 순간부터 유효한 수량 범위를 만족해야 합니다. 현재 API 요청 DTO는 수량 범위를 검증하지만, 도메인 객체 생성자는 직접 생성되는 잘못된 수량을 막지 않을 수 있으므로, `Option` 생성자가 수량 범위를 직접 검증하도록 개선합니다.

**우선순위 이유**: 도메인 객체는 API 입력 경로 외부에서 생성될 수 있습니다. 수량 불변식을 도메인 내부에 두면 테스트, 서비스, 배치, 다른 도메인 흐름에서 직접 생성하더라도 잘못된 상태의 `Option`이 만들어지지 않습니다.

**독립적 테스트**: 잘못된 수량으로 `Option`을 생성하면 예외가 발생하고, 정상 범위 수량으로 생성하면 기존처럼 생성되는지 검증합니다.

**승인 시나리오**:

1. **Given** 수량이 0 이하일 때, **When** `Option`을 생성하면, **Then** 수량 검증 예외가 발생합니다.
2. **Given** 수량이 99,999,999를 초과할 때, **When** `Option`을 생성하면, **Then** 수량 검증 예외가 발생합니다.
3. **Given** 수량이 1 이상 99,999,999 이하일 때, **When** `Option`을 생성하면, **Then** 기존과 동일하게 옵션이 생성됩니다.

---

### 사용자 시나리오 2 - Option 수량 차감 시 차감 수량 검증 (우선순위: P2)

Option 재고를 차감할 때 차감 수량은 1 이상이어야 하며, 현재 재고보다 클 수 없습니다. 현재 재고보다 큰 수량 차감은 이미 막고 있으므로, 0 이하 차감 수량도 도메인에서 명시적으로 막도록 개선합니다.

**우선순위 이유**: 0 또는 음수 차감은 재고 차감이라는 도메인 동작과 맞지 않습니다. 특히 음수 차감은 재고를 증가시키는 버그가 될 수 있으므로 도메인 메서드에서 방어해야 합니다.

**독립적 테스트**: `subtractQuantity`에 0 이하 수량을 전달하면 예외가 발생하고, 현재 재고보다 큰 수량 차감은 기존처럼 예외가 발생하며, 정상 차감은 수량을 감소시키는지 검증합니다.

**승인 시나리오**:

1. **Given** 차감 수량이 0 이하일 때, **When** `subtractQuantity`를 호출하면, **Then** 수량 검증 예외가 발생합니다.
2. **Given** 차감 수량이 현재 재고보다 클 때, **When** `subtractQuantity`를 호출하면, **Then** 기존과 동일하게 재고 부족 예외가 발생합니다.
3. **Given** 차감 수량이 1 이상이고 현재 재고 이하일 때, **When** `subtractQuantity`를 호출하면, **Then** 옵션 수량이 차감 수량만큼 감소합니다.

---

### 사용자 시나리오 3 - 기존 API 입력 검증 유지 (우선순위: P3)

Option 도메인 검증을 추가하더라도 `OptionRequest`의 Bean Validation과 기존 API 응답 계약은 유지합니다.

**우선순위 이유**: API 입력 검증은 클라이언트 요청을 빠르게 검증하고, 도메인 검증은 내부 상태를 보호하는 역할입니다. 두 검증은 서로 대체 관계가 아니라 보완 관계입니다.

**독립적 테스트**: 기존 Option controller/service 테스트가 동일하게 통과하는지 확인합니다.

**승인 시나리오**:

1. Option 생성 API의 수량 Bean Validation은 기존처럼 유지됩니다.
2. Option 이름 검증 흐름은 기존처럼 `OptionNameValidator`와 `OptionValidationException`을 사용합니다.
3. 기존 Option API의 성공/실패 응답 계약은 변경되지 않습니다.

---

### 엣지 케이스

- `quantity`가 0이면 `Option` 생성이 실패해야 합니다.
- `quantity`가 음수이면 `Option` 생성이 실패해야 합니다.
- `quantity`가 99,999,999를 초과하면 `Option` 생성이 실패해야 합니다.
- `subtractQuantity(0)`은 실패해야 합니다.
- `subtractQuantity`에 음수를 전달해도 재고가 증가하면 안 됩니다.
- 현재 재고보다 큰 수량 차감은 기존처럼 실패해야 합니다.
- 옵션명 검증은 이번 작업 범위에 포함하지 않습니다.

## 요구사항 *(필수)*

### 기능 요구사항

- **FR-001**: `Option` 생성자는 수량이 1 이상 99,999,999 이하인지 검증해야 합니다.
- **FR-002**: `Option.subtractQuantity`는 차감 수량이 1 이상인지 검증해야 합니다.
- **FR-003**: `Option.subtractQuantity`는 현재 재고보다 큰 수량 차감을 기존처럼 허용하지 않아야 합니다.
- **FR-004**: `OptionRequest`의 수량 Bean Validation은 유지해야 합니다.
- **FR-005**: 옵션명 검증 흐름은 이번 작업에서 변경하지 않아야 합니다.
- **FR-006**: 수량 도메인 검증을 검증하는 `Option` 단위 테스트를 추가해야 합니다.
- **FR-007**: 기존 Option API의 성공/실패 응답 계약은 변경하지 않아야 합니다.

### 주요 엔티티

- **Option**: 수량 불변식을 직접 검증해야 하는 도메인 엔티티입니다.
- **OptionRequest**: API 입력 단계에서 수량 Bean Validation을 수행하는 요청 DTO입니다.
- **OptionService**: Option 생성 시 기존 요청 검증 및 중복 검증 흐름을 유지합니다.

## 성공 기준 *(필수)*

### 측정 가능한 결과

- **SC-001**: 잘못된 수량으로 `Option` 생성 시 예외가 발생합니다.
- **SC-002**: 정상 수량으로 `Option` 생성 시 기존처럼 생성됩니다.
- **SC-003**: 0 이하 차감 수량으로 `subtractQuantity` 호출 시 예외가 발생합니다.
- **SC-004**: 현재 재고보다 큰 수량 차감 시 기존처럼 예외가 발생합니다.
- **SC-005**: 정상 수량 차감 시 `Option.quantity`가 감소합니다.
- **SC-006**: `./gradlew test --tests *Option*`이 통과합니다.
- **SC-007**: 외부 API 응답 계약의 변경이 없습니다.

## 가정사항

- Option 예외 처리 구조는 `002-option-exception-refactor`에서 이미 완료되었습니다.
- Option 삭제 검증 조회 최적화는 `003-option-deletion-validation-refactor`에서 이미 완료되었습니다.
- Option 삭제 대상 조회 리팩토링은 `004-option-product-scoped-lookup-refactor`에서 이미 완료되었습니다.
- 이번 작업은 수량 도메인 검증만 다루며, 옵션명 검증의 도메인 이동은 포함하지 않습니다.
- 이번 작업은 예외 타입 통일을 포함하지 않으며, 도메인 수량 검증 실패는 우선 `IllegalArgumentException` 기반으로 처리합니다.
- 데이터베이스 schema 변경은 필요하지 않습니다.
