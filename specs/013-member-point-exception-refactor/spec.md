# 기능 명세서: Member 포인트 예외 정리 리팩토링

**Feature Branch**: `013-member-point-exception-refactor`  
**작성일**: 2026-05-18  
**상태**: 초안  
**입력**: "포인트 로직 예외 정리"

## 사용자 시나리오 및 테스트 *(필수)*

### 사용자 시나리오 1 - 포인트 충전 금액 오류를 도메인 예외로 표현 (우선순위: P1)

회원 포인트 충전 금액은 1 이상이어야 합니다. `Member.chargePoint()`는 0 이하 금액이 전달되면 범용 `IllegalArgumentException`이 아니라 member 포인트 도메인 예외를 발생시켜야 합니다.

**우선순위 이유**: 포인트 충전 금액 오류는 회원 포인트 도메인의 명확한 실패 상황입니다. 범용 예외를 사용하면 호출자가 어떤 정책 위반인지 구분하기 어렵습니다.

**독립적 테스트**: 0 이하 금액으로 포인트 충전을 시도하면 `InvalidMemberPointAmountException`이 발생하고 기존 포인트가 유지되는지 검증합니다.

**승인 시나리오**:

1. **Given** 회원 포인트가 0일 때, **When** 1000포인트를 충전하면, **Then** 포인트는 1000이 됩니다.
2. **Given** 회원 포인트가 0일 때, **When** 0 이하 금액을 충전하면, **Then** 포인트 금액 예외가 발생합니다.
3. **Then** 충전 실패 시 기존 포인트는 변경되지 않습니다.

---

### 사용자 시나리오 2 - 포인트 차감 금액 오류를 도메인 예외로 표현 (우선순위: P1)

회원 포인트 차감 금액은 1 이상이어야 합니다. `Member.deductPoint()`는 0 이하 금액이 전달되면 member 포인트 도메인 예외를 발생시켜야 합니다.

**우선순위 이유**: 주문 결제 등 포인트 차감 flow에서 잘못된 차감 금액은 명확한 정책 위반입니다. 충전과 같은 예외 타입으로 처리하면 포인트 금액 정책이 일관됩니다.

**독립적 테스트**: 0 이하 금액으로 포인트 차감을 시도하면 `InvalidMemberPointAmountException`이 발생하고 기존 포인트가 유지되는지 검증합니다.

**승인 시나리오**:

1. **Given** 회원 포인트가 1000일 때, **When** 500포인트를 차감하면, **Then** 포인트는 500이 됩니다.
2. **Given** 회원 포인트가 1000일 때, **When** 0 이하 금액을 차감하면, **Then** 포인트 금액 예외가 발생합니다.
3. **Then** 차감 실패 시 기존 포인트는 변경되지 않습니다.

---

### 사용자 시나리오 3 - 포인트 부족을 도메인 예외로 표현 (우선순위: P1)

회원이 보유한 포인트보다 큰 금액을 차감하려고 하면 `InsufficientMemberPointException`이 발생해야 합니다.

**우선순위 이유**: 포인트 부족은 금액 형식 오류와 다른 실패 상황입니다. 별도 예외 타입으로 분리하면 order 결제 flow에서 의미를 명확히 표현할 수 있습니다.

**독립적 테스트**: 보유 포인트보다 큰 금액을 차감하면 포인트 부족 예외가 발생하고 기존 포인트가 유지되는지 검증합니다.

**승인 시나리오**:

1. **Given** 회원 포인트가 1000일 때, **When** 1500포인트를 차감하면, **Then** 포인트 부족 예외가 발생합니다.
2. **Then** 포인트 부족으로 차감 실패 시 기존 포인트는 변경되지 않습니다.

---

### 사용자 시나리오 4 - 호출 흐름의 동작 유지 (우선순위: P2)

`AdminMemberController`의 포인트 충전 호출과 `OrderController`의 포인트 차감 호출은 기존 메서드를 계속 사용해야 합니다. 이번 작업은 포인트 도메인 예외 타입 정리에 집중합니다.

**우선순위 이유**: admin/order controller의 에러 응답 또는 화면 처리 정책까지 동시에 바꾸면 작업 범위가 커집니다. 우선 도메인 예외를 명확히 한 뒤, controller 처리 방식은 후속 작업에서 정리할 수 있습니다.

**독립적 테스트**: 기존 member/order 관련 테스트 또는 컴파일 검증이 통과하는지 확인합니다.

---

### 사용자 시나리오 5 - 포인트 예외 메시지 한글 통일 (우선순위: P2)

포인트 로직에서 발생하는 예외 메시지는 모두 한글로 제공되어야 합니다. 현재 충전 금액 오류 메시지는 영어이고 차감/부족 메시지는 한글이므로, 사용자와 운영자가 이해하기 쉬운 한글 메시지로 통일합니다.

**우선순위 이유**: 같은 member 포인트 도메인 안에서 예외 메시지 언어가 섞이면 API 응답, 로그, 관리자 화면에서 일관성이 떨어집니다.

**독립적 테스트**: 포인트 금액 오류와 포인트 부족 예외 메시지가 모두 한글인지 검증합니다.

**승인 시나리오**:

1. **Given** 0 이하 금액을 충전할 때, **When** 예외가 발생하면, **Then** 예외 메시지는 `"포인트 금액은 1 이상이어야 합니다."`입니다.
2. **Given** 0 이하 금액을 차감할 때, **When** 예외가 발생하면, **Then** 예외 메시지는 `"포인트 금액은 1 이상이어야 합니다."`입니다.
3. **Given** 보유 포인트보다 큰 금액을 차감할 때, **When** 예외가 발생하면, **Then** 예외 메시지는 `"포인트가 부족합니다."`입니다.

---

### 엣지 케이스

- `chargePoint()`와 `deductPoint()`에 `IllegalArgumentException`이 남지 않아야 합니다.
- 0 이하 금액 오류와 포인트 부족 오류는 서로 다른 예외 타입으로 표현해야 합니다.
- 포인트 예외 메시지는 모두 한글이어야 합니다.
- 예외 발생 시 포인트가 부분 변경되지 않아야 합니다.
- AdminMemberController의 redirect/flash 처리 개선은 이번 작업 범위에 포함하지 않습니다.
- OrderController의 서비스 분리 또는 예외 응답 정리는 이번 작업 범위에 포함하지 않습니다.

## 요구사항 *(필수)*

### 기능 요구사항

- **FR-001**: `Member.chargePoint()`는 0 이하 금액을 허용하지 않아야 합니다.
- **FR-002**: `Member.chargePoint()`의 금액 오류는 `InvalidMemberPointAmountException`으로 표현해야 합니다.
- **FR-003**: `Member.deductPoint()`는 0 이하 금액을 허용하지 않아야 합니다.
- **FR-004**: `Member.deductPoint()`의 금액 오류는 `InvalidMemberPointAmountException`으로 표현해야 합니다.
- **FR-005**: `Member.deductPoint()`는 보유 포인트보다 큰 금액을 허용하지 않아야 합니다.
- **FR-006**: 포인트 부족은 `InsufficientMemberPointException`으로 표현해야 합니다.
- **FR-007**: 포인트 예외는 `MemberException` 하위 타입이어야 합니다.
- **FR-008**: 포인트 예외 발생 시 기존 포인트가 유지되어야 합니다.
- **FR-009**: Member 포인트 도메인 테스트를 추가해야 합니다.
- **FR-010**: 포인트 예외 메시지는 모두 한글로 통일해야 합니다.

### 주요 엔티티

- **Member**: 포인트 충전/차감 정책을 직접 보호합니다.
- **InvalidMemberPointAmountException**: 포인트 충전/차감 금액이 1 미만인 상황을 표현합니다.
- **InsufficientMemberPointException**: 보유 포인트보다 큰 금액을 차감하려는 상황을 표현합니다.
- **AdminMemberController**: 포인트 충전 호출자이며 이번 작업에서는 호출 구조를 변경하지 않습니다.
- **OrderController**: 포인트 차감 호출자이며 이번 작업에서는 호출 구조를 변경하지 않습니다.
