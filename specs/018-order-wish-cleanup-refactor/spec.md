# 기능 명세서: Order 위시리스트 정리 리팩토링

**Feature Branch**: `018-order-wish-cleanup-refactor`  
**작성일**: 2026-05-20  
**상태**: 초안  
**입력**: "주문 성공 시 위시리스트 정리"

## 사용자 시나리오 및 테스트 *(필수)*

### 사용자 시나리오 1 - 주문 성공 시 위시리스트에 있던 상품 삭제 (우선순위: P1)

회원이 위시리스트에 담아둔 상품을 주문하면, 주문 성공 후 해당 상품은 위시리스트에서 자동으로 제거되어야 합니다. 삭제 기준은 주문 옵션의 상품 ID와 주문 회원 ID입니다.

**우선순위 이유**: 위시리스트는 구매 전 관심 상품 목록입니다. 실제 주문이 성공했는데도 동일 상품이 위시리스트에 남아 있으면 사용자에게 완료되지 않은 작업처럼 보일 수 있습니다.

**독립적 테스트**: 위시리스트에 있는 상품의 옵션을 주문하면 `WishRepository.findByMemberIdAndProductId(memberId, productId)`로 조회한 wish가 삭제되는지 검증합니다.

**승인 시나리오**:

1. **Given** 회원의 위시리스트에 상품이 있을 때, **When** 해당 상품의 옵션으로 주문을 생성하면, **Then** 주문은 저장됩니다.
2. **Then** `memberId + productId` 기준으로 찾은 wish가 삭제됩니다.
3. **Then** 주문 생성 응답은 기존처럼 201 Created와 `OrderResponse`를 반환합니다.

---

### 사용자 시나리오 2 - 위시리스트에 없는 상품 주문 시 아무 작업도 하지 않음 (우선순위: P1)

회원이 위시리스트에 없는 상품을 주문해도 주문 생성은 정상적으로 완료되어야 하며, 위시 삭제 시도는 실패로 이어지면 안 됩니다.

**우선순위 이유**: 모든 주문 상품이 위시리스트를 거치는 것은 아닙니다. 위시가 없는 상황은 정상적인 주문 경로이므로 조용히 무시해야 합니다.

**독립적 테스트**: `findByMemberIdAndProductId()`가 empty를 반환할 때 `delete()`가 호출되지 않고 주문 생성이 성공하는지 검증합니다.

**승인 시나리오**:

1. **Given** 회원의 위시리스트에 주문 상품이 없을 때, **When** 주문을 생성하면, **Then** 주문은 성공합니다.
2. **Then** wish 삭제는 수행되지 않습니다.

---

### 사용자 시나리오 3 - 주문 실패 시 위시리스트를 삭제하지 않음 (우선순위: P1)

옵션 미존재, 재고 부족, 포인트 부족 등으로 주문 생성이 실패하면 위시리스트는 변경되지 않아야 합니다.

**우선순위 이유**: 주문이 실패했는데 위시리스트가 삭제되면 사용자는 다시 상품을 찾아 담아야 합니다. 실패 flow에서 사용자 의도를 보존해야 합니다.

**독립적 테스트**: 주문 생성 실패 케이스에서 `WishRepository.findByMemberIdAndProductId()`와 `delete()`가 호출되지 않는지 검증합니다.

**승인 시나리오**:

1. **Given** 존재하지 않는 옵션 ID가 있을 때, **When** 주문 생성을 시도하면, **Then** wish cleanup은 수행되지 않습니다.
2. **Given** 재고가 부족할 때, **When** 주문 생성을 시도하면, **Then** wish cleanup은 수행되지 않습니다.
3. **Given** 포인트가 부족할 때, **When** 주문 생성을 시도하면, **Then** wish cleanup은 수행되지 않습니다.

---

### 사용자 시나리오 4 - 주문 생성 책임 흐름 유지 (우선순위: P2)

위시리스트 정리는 주문 저장 이후, 주문 알림 발송 이전에 수행합니다. 주문 생성의 기존 성공/실패 응답과 알림 서비스 분리 구조는 유지해야 합니다.

**우선순위 이유**: 위시 cleanup은 주문 성공 후 수행되는 부가 정리 작업입니다. 주문 저장이 성공한 뒤 수행해야 하며, 알림 발송보다 먼저 수행하면 주문 트랜잭션 안의 데이터 정리 순서가 명확합니다.

**독립적 테스트**: 주문 저장 후 wish cleanup과 알림 서비스 호출이 모두 수행되는지 검증합니다.

---

### 엣지 케이스

- 삭제 기준은 `memberId + option.product.id`입니다.
- 위시리스트에 주문 상품이 없으면 아무 작업도 하지 않습니다.
- 주문 저장 전 실패하면 wish cleanup을 수행하지 않습니다.
- 주문 성공 응답, 옵션 미존재 404, 포인트 부족 400, 재고 부족 400 응답은 유지합니다.
- 카카오 알림 발송 책임 분리 구조는 유지합니다.
- 위시 cleanup 실패 처리 정책은 이번 작업에서 별도로 정의하지 않습니다.

## 요구사항 *(필수)*

### 기능 요구사항

- **FR-001**: 주문 생성 성공 후 주문 상품이 위시리스트에 있으면 해당 wish를 삭제해야 합니다.
- **FR-002**: wish 조회 기준은 `memberId + productId`여야 합니다.
- **FR-003**: `productId`는 주문한 `Option`의 `Product` ID를 사용해야 합니다.
- **FR-004**: 위시리스트에 해당 상품이 없으면 삭제 없이 주문 성공을 유지해야 합니다.
- **FR-005**: 주문 옵션 미존재 시 wish cleanup을 수행하지 않아야 합니다.
- **FR-006**: 재고 부족 또는 포인트 부족으로 주문이 실패하면 wish cleanup을 수행하지 않아야 합니다.
- **FR-007**: 주문 저장 후 wish cleanup을 수행해야 합니다.
- **FR-008**: wish cleanup 이후 기존처럼 주문 알림 서비스를 호출해야 합니다.
- **FR-009**: 주문 생성 API의 기존 201/400/404 응답 계약을 유지해야 합니다.
- **FR-010**: wish cleanup 정책을 검증하는 service/controller 테스트를 추가 또는 수정해야 합니다.

### 주요 엔티티

- **OrderService**: 주문 저장 후 위시리스트 cleanup을 수행하고 알림 서비스를 호출합니다.
- **WishRepository**: `findByMemberIdAndProductId(memberId, productId)`로 cleanup 대상 wish를 조회하고 삭제합니다.
- **Option**: 주문 옵션이며 cleanup 기준이 되는 상품 정보를 제공합니다.
- **OrderNotificationService**: wish cleanup 이후 주문 완료 알림을 발송합니다.

## 성공 기준 *(필수)*

### 측정 가능한 결과

- **SC-001**: 주문 상품이 위시리스트에 있으면 주문 성공 후 wish가 삭제됩니다.
- **SC-002**: 주문 상품이 위시리스트에 없으면 wish 삭제 없이 주문이 성공합니다.
- **SC-003**: 주문 실패 시 wish 조회/삭제가 수행되지 않습니다.
- **SC-004**: wish cleanup 기준은 `memberId + productId`입니다.
- **SC-005**: 주문 생성 성공/실패 API 응답 계약이 유지됩니다.
- **SC-006**: `./gradlew test --tests *Order* --tests *Wish*`가 통과합니다.
