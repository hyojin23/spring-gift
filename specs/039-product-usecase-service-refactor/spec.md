# 기능 명세서: Product 유스케이스 서비스 공통화 리팩토링

**Feature Branch**: `039-product-usecase-service-refactor`  
**작성일**: 2026-05-23  
**상태**: 초안  
**입력**: "API 상품 서비스와 관리자 상품 서비스 중복 정리"

## 사용자 시나리오 및 테스트 *(필수)*

### 사용자 시나리오 1 - 상품 생성 핵심 흐름 공통화 (우선순위: P1)

개발자는 API 상품 생성과 관리자 상품 생성이 같은 Product 생성/저장 유스케이스를 사용한다고 이해할 수 있어야 합니다.

**우선순위 이유**: `ProductService.createProduct()`와 `AdminProductService.createProduct()`는 카테고리 조회, `Product` 생성, repository 저장 흐름이 겹칩니다. 생성 규칙이 바뀔 때 두 서비스를 모두 수정해야 하는 구조는 유지보수 비용을 키웁니다.

**독립적 테스트**: API 상품 생성 테스트와 관리자 상품 생성 테스트가 기존 응답을 유지하면서 통과해야 합니다.

**승인 시나리오**:

1. **Given** API에서 유효한 상품 생성 요청이 있을 때, **When** 상품을 생성하면, **Then** 기존처럼 `ProductResponse`를 반환합니다.
2. **Given** 관리자 화면에서 유효한 상품 생성 요청이 있을 때, **When** 상품을 생성하면, **Then** 기존처럼 `/admin/products`로 redirect됩니다.
3. **Given** 존재하지 않는 categoryId가 있을 때, **When** 관리자 상품 생성을 요청하면, **Then** 기존처럼 flash error와 함께 `/admin/products`로 redirect됩니다.

---

### 사용자 시나리오 2 - 상품 수정 핵심 흐름 공통화 (우선순위: P1)

개발자는 API 상품 수정과 관리자 상품 수정이 같은 Product 조회/update/저장 유스케이스를 사용한다고 이해할 수 있어야 합니다.

**우선순위 이유**: 수정 흐름도 두 서비스에 중복되어 있습니다. 상품 도메인 수정 방식이 바뀌면 중복 코드가 회귀 위험을 만듭니다.

**독립적 테스트**: API 상품 수정 테스트와 관리자 상품 수정 테스트가 기존 응답을 유지하면서 통과해야 합니다.

---

### 사용자 시나리오 3 - API와 관리자 상품명 정책 차이 유지 (우선순위: P1)

API 상품 생성/수정은 기존처럼 `"카카오"` 포함 상품명을 제한하고, 관리자 상품 생성/수정은 기존처럼 `"카카오"` 포함 상품명을 허용해야 합니다.

**우선순위 이유**: 핵심 저장 흐름은 공통화하더라도 진입 경로별 정책 차이는 유지되어야 합니다.

**독립적 테스트**: 기존 API 상품명 검증 테스트와 관리자 카카오 상품명 허용 테스트가 통과해야 합니다.

---

### 사용자 시나리오 4 - 관리자 화면 모델 보조 역할 유지 (우선순위: P2)

관리자 화면에서 필요한 상품 목록, 카테고리 목록, edit form용 상품 조회는 기존처럼 사용할 수 있어야 합니다.

**우선순위 이유**: 이번 작업은 핵심 생성/수정 유스케이스 중복 제거이며, 관리자 화면 모델 구성 자체를 대규모로 바꾸는 작업이 아닙니다.

**독립적 테스트**: 기존 `AdminProductControllerTest`가 통과해야 합니다.

---

### 엣지 케이스

- 관리자 상품명 검증의 `allowKakao=true` 정책은 유지합니다.
- API 상품명 검증의 `allowKakao=false` 정책은 유지합니다.
- 관리자 상품/카테고리 미존재 예외 메시지와 redirect 정책은 유지합니다.
- API 상품/카테고리 미존재 error code는 유지합니다.
- delete 흐름 공통화는 이번 작업에서 무리하게 포함하지 않고, 필요 시 후속 작업으로 분리합니다.
- `ProductRequest` DTO 구조는 변경하지 않습니다.

## 요구사항 *(필수)*

### 기능 요구사항

- **FR-001**: 상품 생성의 Product 생성/저장 흐름은 공통 유스케이스 메서드로 모아야 합니다.
- **FR-002**: 상품 수정의 Product 조회/update/저장 흐름은 공통 유스케이스 메서드로 모아야 합니다.
- **FR-003**: API 상품 생성/수정은 기존처럼 `ProductResponse`를 반환해야 합니다.
- **FR-004**: 관리자 상품 생성/수정은 기존처럼 view redirect 흐름을 유지해야 합니다.
- **FR-005**: API 상품명 검증 정책은 기존과 동일해야 합니다.
- **FR-006**: 관리자 상품명 검증 정책은 기존과 동일해야 합니다.
- **FR-007**: 관리자 상품 미존재 예외는 기존 `AdminProductNotFoundException` 응답 흐름을 유지해야 합니다.
- **FR-008**: 관리자 카테고리 미존재 예외는 기존 `AdminProductCategoryNotFoundException` 응답 흐름을 유지해야 합니다.
- **FR-009**: API 상품 미존재/카테고리 미존재 예외 응답은 기존과 동일해야 합니다.
- **FR-010**: 공통화 후에도 `ProductServiceTest`, `AdminProductControllerTest`, `ProductControllerTest`가 통과해야 합니다.

### 주요 엔티티

- **ProductService**: API 상품 조회/응답 변환과 상품 핵심 유스케이스를 담당합니다.
- **AdminProductService**: 관리자 화면에 필요한 상품/카테고리 조회와 관리자 정책 진입점을 담당합니다.
- **ProductNameValidator**: 진입 경로별 상품명 정책을 검증합니다.
- **ProductRepository**: 상품 저장소입니다.
- **CategoryRepository**: 카테고리 저장소입니다.

## 가정사항

- 현재 `ProductService`와 `AdminProductService`는 생성/수정 저장 흐름이 겹치지만 상품명 정책은 다릅니다.
- 이번 작업은 중복 제거의 첫 단계로 생성/수정 유스케이스 공통화에 집중합니다.
- 관리자 화면 모델 구성 전체를 재설계하거나 admin service를 제거하는 작업은 후속 단계로 분리합니다.
